# MIG4001 — System.Drawing.Common on non-Windows

| | |
| --- | --- |
| **Category** | Runtime failures |
| **Severity** | High |
| **Effort** | Medium |
| **Confidence tier** | 2 — Probable |

## What it detects

A `using System.Drawing;` (or nested `System.Drawing.*`) directive.

## Why it matters

Since .NET 6, `System.Drawing.Common` is supported only on Windows and throws
`PlatformNotSupportedException` on other platforms. Code compiles fine and fails at runtime
on Linux — the highest-value category of migration surprise.

## How to fix it

If the app must run on Linux, switch to a cross-platform imaging library such as
**ImageSharp** or **SkiaSharp**. If it will only ever run on Windows, you can keep
`System.Drawing.Common` and suppress this rule.

## Known false positives

Tier 2. Fires on a `System.Drawing` reference (a `using` directive or a fully-qualified name);
a file that imports `System.Drawing` but only uses Windows-safe types will still be flagged.
It is also a non-issue for Windows-only deployments.

## Target platform

This is a **Windows lock-in**, not a gone-everywhere API. When you scan with a Windows target
(`--target net10.0-windows`), MigrationScan treats this finding as *satisfied*: it stays listed
for completeness but is excluded from the counts, effort estimate, and `--fail-on`. It only
becomes migration cost if the app must also run off Windows.
