# MIG1007 — Legacy project type

| | |
| --- | --- |
| **Category** | Project and build |
| **Severity** | High |
| **Effort** | Large |
| **Confidence tier** | 1 — Certain |

## What it detects

A project in the solution whose type has no modern .NET equivalent, identified from the `.sln`
entry — by file extension (`.rptproj` SSRS, `.dtproj` SSIS, `.vdproj` setup/installer) or by
project-type GUID (Silverlight, ASP.NET Web Site).

This is detected at the **solution level**, not per project: these aren't C#/VB projects, so
they have no source for the other rules to read. The finding is produced straight from the
`.sln`.

## Why it matters

These project types are dead ends for a code migration — they can't simply be retargeted, and
each needs re-platforming to a supported technology. Left unnoticed, they are the parts of a
solution that quietly blow up a migration estimate.

## How to fix it

| Legacy type | Modern path |
| --- | --- |
| SSRS (`.rptproj`) | Power BI, or a reporting service / library |
| SSIS (`.dtproj`) | Azure Data Factory, or a code-based ETL |
| Setup (`.vdproj`) | WiX or MSIX |
| Silverlight | Blazor or a modern web front end |
| ASP.NET Web Site | Convert to an SDK-style web project |

## Related

Other non-C#/VB projects the scan can't analyze (e.g. `.sqlproj` database, `.deployproj`
deployment) aren't legacy, but they're still listed as **"not assessed"** so their migration is
planned rather than assumed covered. This is a first-class part of every report: a
`projectsNotAssessed` count in the summary, a `notAssessed` array in the JSON, a
"Not assessed" section in the Markdown/console, and a run notification in SARIF.

## Known false positives

None. The project type is read directly from the solution.
