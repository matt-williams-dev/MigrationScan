# MIG1006 — COM reference or interop assembly

| | |
| --- | --- |
| **Category** | Project and build |
| **Severity** | Medium |
| **Effort** | Medium |
| **Confidence tier** | 1 — Certain |

## What it detects

A `<COMReference>` in the project file — a COM type library the project imports as an
interop assembly (e.g. a scanner SDK, an Office automation library, an ActiveX control).

```xml
<!-- Flagged: COM interop -->
<COMReference Include="RANGERLib">
  <Guid>{8b3a1e60-0000-0000-0000-000000000001}</Guid>
  <WrapperTool>tlbimp</WrapperTool>
</COMReference>
```

## Why it matters

This is a **Windows lock-in**, not a gone-everywhere API. COM interop is fully supported on
modern .NET — but only when the project targets Windows (`net10.0-windows`). Off Windows there
is no COM, so the component has no successor and must be replaced.

There is a valid migration path: if the application does not need to run cross-platform,
target `net-windows` and regenerate the interop assembly there. Only if you require Linux/macOS
support does the COM component become a blocker that must be re-architected.

## How to fix it

- **Staying on Windows:** target `net10.0-windows` and regenerate the interop assembly. Very
  old or ActiveX components may still need updating even on Windows.
- **Going cross-platform:** replace the COM component with a managed or cross-platform
  library, or isolate it behind a Windows-only service that the rest of the app calls.

## Known false positives

None on detection — a `<COMReference>` is unambiguous. The *effort* depends on your target:
near-zero if you stay on `net-windows`, potentially large if you must go cross-platform.
