# MIG8003 — Code-page encoding without provider registration

| | |
| --- | --- |
| **Category** | Globalization and encoding |
| **Severity** | Medium |
| **Effort** | Small |
| **Confidence tier** | 2 — Probable |

## What it detects

A `Encoding.GetEncoding(...)` call whose argument is a code page (a number, or a non-Unicode
name). Calls for always-available Unicode encodings (`utf-8`, `utf-16`, `unicode`, `ascii`,
…) are **not** flagged.

## Why it matters

.NET Framework registers legacy code-page encodings by default; modern .NET does not. A call
like `Encoding.GetEncoding(1252)` throws `NotSupportedException` at runtime unless the code
page provider has been registered.

## How to fix it

Register the provider once at startup:

```csharp
Encoding.RegisterProvider(CodePagesEncodingProvider.Instance);
```

(The `System.Text.Encoding.CodePages` package supplies it.) Or move to a Unicode encoding if
the legacy code page is not actually required.

## Known false positives

Tier 2. The rule can't see whether the provider was registered elsewhere, so a codebase that
already registers it will still be flagged. It also cannot resolve a code page passed via a
variable — only literal arguments are classified; a non-literal argument is flagged to be safe.
