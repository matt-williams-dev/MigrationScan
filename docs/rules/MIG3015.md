# MIG3015 — WCF client (System.ServiceModel)

| | |
| --- | --- |
| **Category** | Blocking frameworks |
| **Severity** | Medium |
| **Effort** | Small |
| **Confidence tier** | 2 — Probable |

## What it detects

Use of the `System.ServiceModel` namespace — typically a WCF **client** or a generated service
reference. Reported **once per file**. Auto-generated proxies (e.g. a svcutil `Reference.cs`)
are skipped, so only hand-written usage is flagged.

## Why it matters

Unlike server-side hosting ([MIG3004](MIG3004.md), which is *not* supported on modern .NET),
the WCF **client** is supported — but it isn't in the box. You must add the `System.ServiceModel.*`
client packages, and config-based endpoints usually need moving to code.

## How to fix it

- Add the client packages you need: `System.ServiceModel.Http`, `System.ServiceModel.NetTcp`,
  `System.ServiceModel.Primitives` (and `dotnet-svcutil` to regenerate the proxy).
- Move `<system.serviceModel>` endpoint/binding configuration from `app.config` to code —
  modern .NET does not read the WCF config section.
- Some bindings and behaviors (e.g. WS-* features, MSMQ transport) are unsupported; check each.

## Known false positives

Tier 2. Matches any `System.ServiceModel` reference, so a project that only touches a WCF type
incidentally is still flagged. On a server project this fires alongside MIG3004; the two are
complementary (client packages vs. host rewrite).
