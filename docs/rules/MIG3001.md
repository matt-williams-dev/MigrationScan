# MIG3001 — ASP.NET WebForms

| | |
| --- | --- |
| **Category** | Blocking frameworks |
| **Severity** | Blocker |
| **Effort** | Blocker |
| **Confidence tier** | 1 — Certain |

## What it detects

The presence of `.aspx`, `.ascx`, or `.master` files in the project directory — the
signature of an ASP.NET WebForms application.

## Why it matters

WebForms has no counterpart on modern .NET. There is no migration path that preserves the
page/control model; the UI layer must be re-architected. This is a blocker: it needs an
architectural decision before any estimate is meaningful.

## How to fix it

Plan a re-architecture:

- **Razor Pages** — closest page-centric model, good fit for form-over-data apps.
- **MVC** — if you want explicit controllers.
- **Blazor** — if you want a component/stateful-UI model closer to WebForms' feel.

Expect to rebuild the view layer and replace server controls, ViewState, and the page
lifecycle. Business logic behind the pages is usually more portable.

## Known false positives

None. The presence of WebForms files is unambiguous.
