# MIG1005 — GAC reference (no HintPath)

| | |
| --- | --- |
| **Category** | Project and build |
| **Severity** | Medium |
| **Effort** | Medium |
| **Confidence tier** | 1 — Certain |

## What it detects

A `<Reference>` that is **strong-named** (its `Include` carries a `PublicKeyToken`),
has **no `<HintPath>`**, and is **not a framework assembly**. Such a reference resolves
from the Global Assembly Cache.

```xml
<!-- Flagged: strong-named, no HintPath, third-party -->
<Reference Include="Telerik.Web.UI, Version=2019.1.115.45, Culture=neutral, PublicKeyToken=121fae78165ba3d4" />
```

Framework assemblies (`System.*`, `mscorlib`, `PresentationCore`, …) are excluded — they
are handled by the framework-specific rules, not treated as third-party GAC dependencies.

## Why it matters

The GAC does not exist on modern .NET. A dependency that resolves from the GAC will not
be found at build or run time after migration.

## How to fix it

- Prefer a NuGet package for the dependency, if one exists.
- Otherwise, check the assembly into the repository and add an explicit `<HintPath>`.

## Known false positives

The rule keys on the *form* of the reference (strong-named, no HintPath), which is
certain, but "resolves from the GAC" is an interpretation. A strong-named reference that
happens to resolve via a build-time assembly-search path could be flagged even though it
is not strictly GAC-resolved. The remediation (package or HintPath) is correct either way.
