# MIG1001 — Non-SDK-style project file

| | |
| --- | --- |
| **Category** | Project and build |
| **Severity** | Medium |
| **Effort** | Small |
| **Confidence tier** | 1 — Certain |

## What it detects

A `.csproj` whose root `<Project>` element has no `Sdk` attribute. SDK-style projects
begin with `<Project Sdk="Microsoft.NET.Sdk">` (or another SDK); legacy projects instead
declare an `xmlns` and import `.targets` files explicitly. The absence of the `Sdk`
attribute is read straight from the project XML, so this is a Tier 1 (certain) finding.

```xml
<!-- Legacy (flagged) -->
<Project ToolsVersion="15.0" xmlns="http://schemas.microsoft.com/developer/msbuild/2003">
  ...
</Project>

<!-- SDK-style (not flagged) -->
<Project Sdk="Microsoft.NET.Sdk">
  ...
</Project>
```

## Why it matters

Non-SDK-style projects cannot target modern .NET without conversion. Almost every other
step in a migration — retargeting the framework, switching to `PackageReference`, trimming
the file list — assumes an SDK-style project. Converting first makes the rest of the work
tractable, which is why this is usually the first thing to fix.

## How to fix it

Convert the project to the SDK style:

1. Replace the root element with `<Project Sdk="Microsoft.NET.Sdk">` (or `Microsoft.NET.Sdk.Web`
   for web projects, `Microsoft.NET.Sdk.WindowsDesktop` for WinForms/WPF).
2. Replace `<TargetFrameworkVersion>v4.7.2</TargetFrameworkVersion>` with a target framework
   moniker, e.g. `<TargetFramework>net472</TargetFramework>` initially, then retarget upward.
3. Move `packages.config` entries to `<PackageReference>` (see MIG1002).
4. Delete the explicit `<Compile Include=...>` items and let the SDK glob source files.
5. Remove the boilerplate `.props`/`.targets` imports the SDK now provides.

## Known false positives

None. The presence or absence of the `Sdk` attribute is unambiguous.
