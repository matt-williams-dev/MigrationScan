# MIG3002 — System.Web dependency outside WebForms

| | |
| --- | --- |
| **Category** | Blocking frameworks |
| **Severity** | High |
| **Effort** | Medium |
| **Confidence tier** | 1 — Certain |

## What it detects

A `<Reference Include="System.Web" />` in a project that is **not** a WebForms application
(no `.aspx`/`.ascx`/`.master` files — those are reported by [MIG3001](MIG3001.md) instead).

## Why it matters

`System.Web` is not available on modern .NET. Class libraries and services that reach into
`System.Web` — for `HttpContext`, `HttpUtility`, `HttpRuntime`, and similar — will not
compile or run after migration.

## How to fix it

Replace the `System.Web` dependency:

- `HttpContext` → `Microsoft.AspNetCore.Http.HttpContext` (via `IHttpContextAccessor`).
- `HttpUtility.HtmlEncode` / `UrlEncode` → `System.Net.WebUtility` or `System.Web.HttpUtility`
  from the `System.Web.HttpUtility`-equivalent APIs.
- `HttpRuntime.Cache` → `Microsoft.Extensions.Caching.Memory`.

## Known false positives

Low. The `System.Web` assembly reference is read directly from the project file. A project
that references `System.Web` purely for a type that has a trivial modern equivalent will
still be flagged — which is correct, since the reference itself must be removed.
